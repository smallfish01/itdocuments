Exchange 2013 设置代理邮件发送

####需求
用户希望以运营部(op@domain.com)身份给其他人发送邮件，由于该账号为邮件组，非单独用户，无法通过密码登录Web或邮件客户端方式发送，需要配置代理或代表发送，本例以用户(userA@domain.com)代理运营部(op@domain)外发邮件。

####实施方法

代表(Send-On-Behalf)和代理(Send As)主要区别如下：

代表发送时会显示代表人邮件地址，例如用户A代表用户B发送邮件，当用户收到邮件时会被告知该邮件是由用户A代表B发送的。如果是代理发送，收件人不知道谁代理人，对用户来说是不知道真正发件人的。

####配置

1.登录Windows AD Server，在AD用户和计算机的对应OU下找到该邮件组–右键属性–安全–添加用户userA，并赋予权限：”另外发送为“，点击应用确认。


也可以通过命令行方式添加，登录Exchange服务器，在Exchange Management Shell下输入：


或登录ECP 在组委派里添加即可。

2.登录Exchange ECP 确认：

ECP–收件人–组--op@domain.com–双击选择组委派–发送方式。可以看到该用户以被添加。


3.发送测试

登录Webmail,创建新邮件，点击右上角”…”–显示发件人。



删除默认的发件人，输入被代理地址om@doamin.com.





输入邮件内容发送，收件人收到后显示为发件人为:op@domain.com，不会显示代理人地址。


如果要删除该用户代理发送权限，可以在AD用户组安全里删除该用户，或者直接在ECP上删除。

4.代表发送

如果要使用代表发送，可以直接在ECP中选中要被代表的邮件组–组委派–代表发送–添加用户即可，或者通过Exchange Management Shell添加。


5.如果要让某用户用代理或代表某个用户(非邮件组)发送邮件，也可以使用以上方法操作，对应ECP 设置区域为”邮箱委托“。

Posted on 2019-09-09 | In Exchange&Outlook | Visitors:

邮件服务器双机高可用方案

背景

之前邮件系统由一台物理机构成，部署两个VM，包括邮件网关和Exchange.由于所有服务都运行在一台物理机上，如果有硬件或系统问题将会导致邮件系统不可用。

目标

为解决单点故障问题，需要将当前的Exchange server All in one模式拆分成CAS和MBX服务器分离，拆分部署在两台物理机上，当其中一台物理机或系统故障时不会影响整个邮件系统。

前提条件

需要有Windows AD服务器，且所有VM都部署同样版本的操作系统，加入域。

准备Exchange server 和必须的安装软件。

环境信息

Windows AD 和邮件系统操作系统为Windows server 2012 R2标准版。

Exchange server 版本为2013。

CAS VM两台，服务器名为CAS01,CAS02，分别部署在两台物理机上，配置NLB并设置一个单独的域名CAS-NLB和IP。

MBX VM两台, 服务器名为MBX01,MBX02,分别部署在两台物理机上，配置心跳地址和DAG。

1. 基础配置

1).准备4台VM，分别用于CAS和MBX,安装Windows server 2012 R2，加入域并安装必要的补丁;

2).使用Hyper-v自带的复制功能，为邮件网关服务器启用复制，定时将VM数据同步到另外一台物理机，确保当源VM或宿主机故障时，可以手动启用另外一台邮件网关，正常切换时间大约为2~5分钟，确保邮件外发和接收正常。

检查邮箱数据库状态：

get-mailboxdatabase -status | select name,mounted,mountedonserver

1).CAS服务器需要安装NLB(网络负载均衡)服务，只需在其中一台CAS服务器配置即可，NLB地址为nlb-cas.domain.com，配置后确认NLB工作正常；

2).配置完毕后，登录ECP并参考旧服务器配置，修改CAS和MBX配置，包括：CAS证书，虚拟目录，服务器等，对外的CAS域名需要使用nlb负载均衡cas-nlb的地址；

3).检查内网防火墙，DNS服务器以及邮件网关配置的CAS服务器地址，修改对外映射的邮件服务器域名和IP；包括webmail/mail/autodiscover.domain.com域名对应的IP，修改后测试邮件收发是否正常。

5).通过ECP对CAS修改完毕后，确认邮件网关配置修改正确，尝试断开旧的Exchange 服务器网络,测试邮件收发是否正常,如果邮件收发正常即代表CAS切换成功。

Windows server 2012 Hyper-v 虚拟机复制

目的

利用Hyper-v 自带的复制功能，将日常运行的虚拟机定时复制到远程主机，用于实现当主机故障时，可以手动通过被复制主机继续提供服务，以达到最低成本的服务器故障切换。

本配置目的测试将源Hyper-v(192.168.20.5) 所属的一台VM(192.168.20.51)配置成复制主机模式，每隔5分钟复制VM到目标Hyper-v(192.168.20.7)主机。

测试正常情况下当源VM关闭并临时启用计划的故障转移后，两台服务器之间的切换时间，最后将恢复源主机角色的配置过程。

结论

在被测试VM大小为26G的情况下，两台主机之间初始复制时间为20分钟左右，切换时机器关闭和启动时间在1~3分钟。

需求

源主机和被复制目标主机安装WindowsServer2012 标准版或数据中心版以及Hyper-v。

源主机和目的主机Hyper-v都设置为允许复制。

防火墙设置相互允许http,https访问。


配置概要

源主机配置复制，复制完成后关机，进行计划的故障转移；

启用被复制VM，确认资料数据完整，启用反向复制并关闭该VM；

重新启用源复制VM，确认数据正常，启用反向复制；

切换流程结束。

配置步骤

1.在源Hyper-v所在的VM配置启用复制。

选中要复制的主机，点击右键–启用复制，根据提示，将源主机指定VM 192.168.20.51复制到目标主机192.168.20.7，整个过程根据VM容量和网络速度而定，一般内网26G左右数据需要20分钟左右。


2.启用故障转移。

在源VM下做些修改，例如修改用户密码，从别处拷贝文件，以确定故障转移后文件是否恢复到目标服务器。

关闭VM并选中点击右键复制–计划的故障转移，确定后点击故障转移,整个过程将持续2分钟左右。






3.确认目标VM启动和数据完整。

转移完毕后目标服务器将自动启动，整个过程在1~3分钟。

打开目标VM，确认在故障转移之前的文件和用户密码修改是否有效。此时检查两台VM的复制状态和之前一样，源VM为复制主机，目标VM为被复制主机，因为目前还没做反向复制。

4.反向复制测试。

当启用被复制VM并运行一段时间后可能会产生一些数据，此时为了确保两台服务数据一致，要进行反向复制，已确保源主机数据也保持更新。

在当前运行主机上做一些修改操作，例如创建用户账户，删除文件，操作完毕后进行反向复制，将当前内容复制到目标Hyper-v(182.168.20.5)下的VM。即选中目标VM–右键复制–反向复制，预计持续时间将1~3分钟。


5.恢复源主机。

数据反向复制完成后，关闭当前VM,选择右键复制–计划的故障转移，远程目标主机Hyper-v(192.168.20.5)下的VM将启动，大约从关机到启动需要1~3分钟。


6.确认复制效果。

登录目标VM，检查之前增加的用户名和删除的文件是否生效。

然后选择VM，点击右键复制–反向复制，重新启动正常的复制流程,当前的被复制主机又变成了之前192.168.20.5下的VM。

7.取消复制。

如果要取消复制，需要在源主机和目标主机都取消，否则只取消源主机，目标主机还会继续复制。

Windows Hyper-V 部署复制功能

目的

通过Hyper-v自带的复制功能，在主服务器VM在正常运行的时候，定时自动复制数据到另一台VM，当主服务器故障时，可以很快通过切换到被复制主机恢复服务器运行。

环境

源主机地址：192.168.20.30，所在宿主机：192.168.20.9
被复制主机：192.168.20.30，所在书主机：192.168.20.8

配置步骤

1.需要配置两台Hyper-v启用复制并设置防火墙允许http/https：

2.选中要复制的虚拟机–邮件–复制，配置复制到另外一台宿主机,复制方法和时间根据实际需求调整，一般选用默认即可。

3.复制配置完成后，备机默认是关闭状态，可以在主备机上查看当前复制状态。

4.当主机故障无法连接的时候，可以启用备机进行恢复模式。选择备机–右键–复制–故障转移。启用备机提供服务。
当主机恢复后，可以关闭备机，重新启用主机提供服务。

5.使用反向复制功能。

当主机不能持续提供服务，需要切换为备机状态时，可以将当前的备机提升为主机，配置反向复制。
将主机关机，选择复制–计划的故障转移–将服务转移到备机上。

服务转移到备机后，在主机上选择反向复制：

根据提示操作即可，将当前备机提升为主机:


配置完毕后在主备机上检查确认即可。
